# План разработки — обзор (35/65)

Работа делится на **2 сессии** по объёму: **35%** в первую, **65%** во вторую.

- **Сессия 1 — Фундамент (~35%)** — каркас, на котором держится всё остальное.
  Приложение запускается, можно залогиниться, роли работают, навигация есть.
  Подробно: [`session-1-foundation.md`](session-1-foundation.md)

- **Сессия 2 — Основная функциональность (~65%)** — рабочее приложение
  по критериям приёмки. Идём по сущностям сверху вниз.
  Подробно: [`session-2-features.md`](session-2-features.md)

## Почему такой баланс

- Сессия 1 меньше по «фичам», но критична: настройка окружения, ролей и
  Security Rules съедает много времени. Если фундамент кривой — вторая сессия встанет.
- Блок «Банк вопросов и квизы» во второй сессии — самый тяжёлый, реально около
  половины её объёма. При нехватке времени сюда в первую очередь идёт MVP-срез
  (4 типа вопросов вместо 12).

## Технические решения (зафиксировать до старта)

- **State management:** Riverpod
- **Роутинг:** go_router
- **Модели:** freezed + json_serializable
- **Backend:** Firebase (Auth, Firestore, Storage, Cloud Functions, FCM)
- **Локальная разработка:** Firebase Emulator Suite

## Ключевые риски (держать в голове)

1. **Объём большой** — обязательно зафиксировать MVP-срез, иначе scope creep.
2. **Поиск в Firestore** — нет полнотекстового поиска; нужен Algolia/Typesense
   или префиксные индексы. Решить заранее.
3. **Агрегированная статистика** — через Cloud Functions + denormalization,
   а не чтение тысяч документов.
4. **Security Rules** — самая частая точка провала Firebase-проектов;
   писать рано и тестировать в эмуляторе.
5. **iOS-сборка** — нужен Apple Developer аккаунт и Mac для билда.
